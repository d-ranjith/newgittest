## **7. Implementation Steps (Updated Sections Only)** ### **Step 2 – Deploy and Configure PostgreSQL** The earlier step creating only a Kubernetes secret is **not sufficient**. Tekton Results requires a **running PostgreSQL instance, schema initialization, and API configuration**. #### 2.1 Deploy PostgreSQL Deploy a PostgreSQL instance accessible from the cluster. Example (POC only): ``` apiVersion: apps/v1 kind: Deployment metadata: name: tekton-results-postgres namespace: tekton-results spec: replicas: 1 selector: matchLabels: app: tekton-results-postgres template: metadata: labels: app: tekton-results-postgres spec: containers: - name: postgres image: postgres:14 env: - name: POSTGRES_USER value: tekton - name: POSTGRES_PASSWORD value: tektonpw - name: POSTGRES_DB value: tekton_results ``` Create a service for the database. ``` kubectl expose deployment tekton-results-postgres \ --name tekton-results-postgres \ --port 5432 \ --target-port 5432 \ -n tekton-results ``` --- ### **2.2 Create Database Credentials Secret** Create credentials used by the Results API. ``` kubectl create secret generic tekton-results-postgres \ --from-literal=POSTGRES_USER=tekton \ --from-literal=POSTGRES_PASSWORD= \ -n tekton-results ``` --- ### **2.3 Install Tekton Results** Install the Results components. ``` kubectl apply -f \ https://storage.googleapis.com/tekton-releases/results/latest/release.yaml ``` Components installed: * tekton-results-api * tekton-results-watcher * tekton-results-retention-policy-agent --- ### **2.4 Configure Results API to Use PostgreSQL** By default Tekton Results uses **SQLite**. It must be configured to use PostgreSQL. Create or patch the configuration: ``` apiVersion: v1 kind: ConfigMap metadata: name: tekton-results-api-config namespace: tekton-results data: db_driver: postgres db_host: tekton-results-postgres db_port: "5432" db_name: tekton_results ``` Apply the configuration and restart the Results API deployment. --- ### **2.5 Initialize Database Schema** Before results can be stored, the database schema must be initialized. Run migration command from the Results API container: ``` kubectl exec -it deployment/tekton-results-api \ -n tekton-results -- results migrate ``` This creates required tables such as: * records * results * logs --- ### **2.6 Verify Results API Connectivity** Check the Results API logs. ``` kubectl logs deployment/tekton-results-api -n tekton-results ``` Expected output should confirm connection to PostgreSQL. Example: ``` Connected to postgres database ``` --- ### **2.7 Validate Watcher Ingestion** Trigger a sample pipeline run. Verify watcher activity: ``` kubectl logs deployment/tekton-results-watcher -n tekton-results ``` Expected log messages: ``` storing PipelineRun result ``` --- ## **Additional Configuration (Recommended)** ### **Use Kustomize Overlay for Database Configuration** When customizing Tekton Results deployments, use **Kustomize overlays**. Example structure: ``` tekton-results/ ├─ base └─ overlays └─ postgres kustomization.yaml api-config-patch.yaml ``` Example overlay configuration: ``` resources: - ../../base patchesStrategicMerge: - api-config-patch.yaml ``` Apply using: ``` kubectl apply -k overlays/postgres ``` --- ## **Production Considerations** For large environments (~1M builds), avoid running PostgreSQL inside the cluster. Recommended options: * Managed PostgreSQL service * HA PostgreSQL cluster * Dedicated storage with backups Benefits: * High availability * Automated backups * Improved scalability * Reduced operational overhead ---
